# 0.6.2 Delivery Plan — Design Spec

**Date**: 2026-06-01
**Status**: Draft for owner review
**Scope**: Phase A type system enhancements + error/performance/tooling passes
**Target**: eucalypt 0.6.2

## 1. Overview

Release 0.6.2 delivers Phase A of the type system evolution — closing
the existing gradual type system so it feels finished for everyday
data-transformation work. Alongside the type system beads, Clarion,
Stopwatch, and Lantern run continuous improvement passes for error
quality, performance, and tooling/documentation.

### What ships

| Area | Beads | Agent |
|------|-------|-------|
| Type system — row polymorphism + Dict | eu-zvmr (A1), eu-g8df (A2) | Quill |
| Type system — recursive types | eu-z9zz.9 (A3) | Quill |
| Type system — literal types | eu-19do (A4) | Quill |
| Type system — flow narrowing | eu-hj08 (A5) | Quill |
| Type system — NonEmpty refinement | eu-2mr2 (A6) | Quill |
| Type system — alias references in DSL | eu-7s1r (A7) | Quill |
| Type system — monadic bound-var hints | eu-oypa (A10) | Quill |
| Error diagnostics pass | TBD | Clarion |
| Performance pass | TBD | Stopwatch |
| Documentation + tooling review | TBD | Lantern |

**Note on A9 (eu-ggr9)**: Monadic block binding type hints shipped in
0.6.0 — the bead is closed with all criteria met. It is NOT 0.6.2 work.
A10 (eu-oypa) is the remaining follow-on that infers the *unwrapped*
element type.

### What does NOT ship (deferred to 0.7.0)

Phase B beads: HKT (eu-crbc), structural constraints (eu-z9zz.8),
Partial(T) (eu-lqzf), dependent access (eu-19ue), prelude cache
(eu-if4x), monad HKT (eu-6kjc), row inference (eu-9jlt).

## 2. Branch Discipline

### Integration branch: `integration/0.6.2`

All agent work targets `integration/0.6.2`, never master directly.

```
master (0.6.1 — stable, released)
  └── integration/0.6.2 (Wicket manages)
        ├── feat/quill-A1-row-poly
        ├── feat/quill-A2-dict-types
        ├── fix/clarion-error-xyz
        ├── perf/stopwatch-xyz
        └── docs/lantern-xyz
```

**Rules:**
- Feature branches fork from `integration/0.6.2`, PRs target it
- Wicket has full authority to merge PRs to `integration/0.6.2`
- Wicket must NOT merge to master — only the owner does this
- Owner merges `integration/0.6.2` → master at release time
- `integration/0.6.2` must stay green — Wicket must not merge when CI is red

### Branch naming

| Agent | Pattern | Example |
|-------|---------|---------|
| Quill | `feat/quill-<bead-slug>` | `feat/quill-A1-row-poly` |
| Clarion | `fix/clarion-<description>` | `fix/clarion-lookup-error-values` |
| Stopwatch | `perf/stopwatch-<description>` | `perf/stopwatch-unifier-hot-path` |
| Lantern | `docs/lantern-<description>` | `docs/lantern-type-guide-update` |

### CI on the integration branch

The CI workflow (`build-rust.yaml`) must be updated to run the full
test matrix on pushes to `integration/0.6.2`:

- **Push triggers**: `[master, integration/0.6.2]`
- **Test matrix on integration pushes**: includes macOS (same as master)
- **aarch64 sequential GC-stress test**: runs on integration pushes
- **ASAN**: runs on integration pushes
- **Doc example tests**: run on integration pushes
- **Release candidate builds**: master only (unchanged)

This ensures Wicket can trust `integration/0.6.2` as a master proxy.

## 3. Agent Roles and Governance

### Quill — Type system implementation

Works sequentially through Phase A beads:

1. **A1 + A2** (row poly + Dict) — adopt together per spec
2. **A3** (recursive types)
3. **A4** (literal types) — starts critical path
4. **A5** (flow narrowing) — blocked by A4
5. **A6** (NonEmpty) — blocked by A5
6. **A7** (alias references)
7. **A10** (monadic bound-variable hints)

Sequential to avoid merge conflicts in `src/core/typecheck/`.

**Before each bead**: Quill must read the spec, the acceptance
criteria, and run existing tests. Must not start implementation until
the previous bead's PR is merged to `integration/0.6.2`.

### Clarion — Error diagnostics (tightly scoped)

Creates a review bead, audits error paths, raises sub-beads for
specific fixes.

**Valuable work (in scope):**
- Including real data in error messages (actual key names, actual
  types, actual values)
- Fixing missing or wrong source locations
- Improving stack traces (better frames, less noise)
- Converting panics to proper errors with diagnostics
- Auditing and REMOVING misleading notes/hints from prior work

**NOT in scope (causes regressions):**
- Adding "helpful" notes or hints to errors — these have been
  misleading in most real-world cases and must stop
- Rewording messages for style
- Adding context relevant to only one specific scenario

Wicket enforces this scope. Any PR that adds notes/hints to error
messages is sent back automatically.

### Stopwatch — Performance

Profiles representative workloads, identifies bottlenecks, proposes
targeted improvements as separate PRs.

**All `eu` processes must use `timeout`.** Agent-reported benchmarks
must be independently verified by Wicket before merge.

### Lantern — Documentation and tooling

**Phase 1** (start of release): Review docs, tree-sitter grammar,
Emacs mode, VS Code extension against current 0.6.1 features. Raise
beads for each issue found.

**Phase 2** (ongoing): After each Quill bead lands on
`integration/0.6.2`, review whether docs need updating for the new
type features. Specific files to check:
- `docs/guide/` — user-facing type system guide
- `docs/reference/agent-reference.md` — agent syntax reference
- `docs/appendices/cheat-sheet.md` — quick reference
- `docs/appendices/syntax-gotchas.md` — pitfalls

### Wicket — Gatekeeper

Reviews all PRs targeting `integration/0.6.2`. Merges or sends back.

**Review process for every PR:**

1. **CI gate** — `gh pr checks` must be green. No exceptions.
2. **Code review** — thorough review using `/review` (superpowers
   code-review skill). Checks correctness, edge cases, test coverage.
3. **Acceptance criteria gate** — read the bead's acceptance criteria
   (from `bd show <id>`). Every criterion must be demonstrated. If any
   are missing, send the PR back with specific gaps listed.
4. **Harness test gate** — the PR must include harness tests that
   exercise the feature. No tests = send back.
5. **Documentation gate** — per existing Wicket doc gate table.
6. **Scope check** — no scope creep beyond the bead.

**Clarion/Stopwatch filter:**
- PRs that are purely internal improvements (better error messages
  with real data, faster hot paths, no API/behaviour change): Wicket
  may merge.
- PRs that change observable behaviour, add new warnings/errors, or
  modify core data structures: flag for owner review. Do NOT merge.
- Any Clarion PR that adds notes/hints to error messages: send back
  automatically. This is out of scope.

**Bead closure:**
- Wicket must not close beads. Wicket merges PRs and confirms
  acceptance criteria are met.
- Beads are closed by the coordinator after Wicket confirms all
  criteria are satisfied.
- No partial closure. No "defer remaining work." The bead is done or
  it isn't.

### Furnace — On standby

Not assigned work unless GC/runtime issues surface during the release.

## 4. Acceptance Criteria

Every type system bead has explicit, testable acceptance criteria.
These are loaded into the bead via `bd update <id> --acceptance="..."`.

### eu-zvmr (A1) — Row polymorphism in inference

1. `merge`/`over` carry row-polymorphic types in their annotations
2. Row content demonstrably flows through merge: `a merge(b)` with
   `a : {x: number}`, `b : {y: string}` synthesises
   `{x: number, y: string}`, not `{..}`
3. `eu check lib/prelude.eu` produces no new warnings
4. Harness tests: row-preserving inference through merge, over, and
   user-defined row-stable functions
5. Existing harness tests remain green

### eu-g8df (A2) — Homogeneous block (Dict) types

1. `Type::Dict(Box<Type>)` variant exists
2. Subtyping: `{a: A, b: B} <: Dict(A|B)` and `Dict(T) <: {..}`
3. `map-values`, `group-by`, `values`, `keys` carry Dict-based types
4. `eu check lib/prelude.eu` produces no new warnings
5. Harness tests: Dict subtyping, widening from record literals,
   prelude function type propagation
6. Existing harness tests remain green

### eu-z9zz.9 (A3) — Recursive types (equirecursive)

1. `Type::Mu` constructor exists
2. Self-referencing type alias resolves to `Mu` form (not `any`)
3. A recursive `Json` alias type-checks
4. Coinductive subtyping terminates (memoised assumed-pairs set)
5. Display is finite (no infinite unfolding)
6. Harness tests: recursive alias definition, subtyping over recursive
   types, display round-trip
7. Existing harness tests remain green

### eu-19do (A4) — Literal types (string & bool)

1. `LiteralString(String)` and `LiteralBool(bool)` type variants exist
2. Subtyping: `LiteralString(v) <: string`, `LiteralBool(v) <: bool`
3. String/bool literals synthesise literal types in the checker
4. Widening at annotation/recursive-call boundaries keeps existing
   tests green
5. Harness tests: literal synthesis, subtyping, widening behaviour
6. `LiteralNumber` explicitly NOT in scope
7. Existing harness tests remain green

### eu-hj08 (A5) — Flow-sensitive narrowing + type subtraction

1. Type predicates (`number?`, `string?`, `nil?`, etc.) narrow types
   in branch arms
2. Null-discrimination via `✓` (not-null) narrows appropriately
3. Union subtraction `T - U` works over union types
4. Structural BranchShape recognition covers `if`, `then`, `cond`,
   `||`, `&&`
5. Rebinding `if` (or other branch combinators) disables narrowing
   without false positives
6. Harness tests: narrowing in each branch of `if`, `cond`;
   subtraction over unions; rebound combinator test
7. Existing harness tests remain green

### eu-2mr2 (A6) — NonEmpty refinement + branch-narrowing

1. `NonEmpty([a])` type constructor exists
2. `[] head` warns; `[1] head` does not
3. Post-`nil?` branch narrows to `NonEmpty`
4. `[k, v] head` types precisely (HEAD/TAIL-on-Tuple element typing)
5. **`eu check lib/prelude.eu` emits zero warnings** — non-negotiable
   gate from the spec
6. Harness tests: empty head warning, non-empty head clean, nil?
   branch narrowing, tuple element typing
7. Existing harness tests remain green

### eu-7s1r (A7) — First-class alias references in the type DSL

1. Type-string parser records source spans for alias identifiers
2. LSP indexes `types:` blocks and `type-def:` declarations into an
   alias-name → definition-site map
3. Go-to-definition works for an alias in a `type:` string
4. Hover shows the alias definition
5. Rename works for aliases
6. Harness tests: N/A (LSP feature — tested via LSP test suite)
7. LSP tests cover go-to-def, hover, and rename for type aliases

### eu-oypa (A10) — Monadic bound-variable element-type hints

1. Bound variables inside monadic blocks synthesise their unwrapped
   element type (e.g. `x` in `{ :for x: [1,2,3] }` is `number`)
2. LSP inlay hints reflect the unwrapped type
3. Harness tests: monadic bound variable typing for `for`, `io`,
   user-defined monads
4. Existing harness tests remain green

## 5. Clarion Scope — Error Diagnostics Pass

### Audit bead (to be created)

Clarion audits all error paths and raises sub-beads for:
- Error messages that lack real data (key names, type names, values)
- Missing or incorrect source locations
- Stack traces with excessive noise or missing useful frames
- Remaining panics in user-reachable code paths
- **Misleading notes/hints from prior Clarion work** — audit and
  remove notes that are irrelevant or confusing in common cases

### Hard constraints

- NO new notes or hints on error messages
- NO rewording for style
- NO context that only applies to one specific scenario
- Every fix must include an error harness test (`.eu` + `.expect`)
- Wicket sends back any PR that violates these constraints

## 8. Sequencing

```
Phase 0: Setup
  ├── Create integration/0.6.2 branch from master
  ├── Update CI workflow for integration branch
  ├── Update agent definitions (Wicket, Clarion, Stopwatch, Lantern)
  ├── Populate acceptance criteria on all beads
  └── Create Clarion/Stopwatch/Lantern audit beads

Phase 1: Parallel kickoff
  ├── Quill: A1 + A2 (row poly + Dict — adopt together)
  ├── Clarion: error audit → raise sub-beads → fix
  ├── Stopwatch: profiling → raise sub-beads → fix
  ├── Lantern: tooling/docs review → raise beads → fix
  └── Wicket: reviewing PRs as they arrive

Phase 2: Quill critical path (Clarion/Stopwatch/Lantern continue)
  ├── A3 (recursive types)
  ├── A4 (literal types)
  ├── A5 (flow narrowing)
  └── A6 (NonEmpty)

Phase 3: Tail work
  ├── A7 (alias references in type DSL)
  ├── A10 (monadic bound-variable hints)
  ├── Lantern: second pass for new type features
  └── Clarion/Stopwatch: wrap up remaining items

Phase 4: Release
  ├── Version bump to 0.6.2 in Cargo.toml
  ├── CHANGELOG.md update
  ├── Owner reviews integration/0.6.2
  ├── Owner merges integration/0.6.2 → master
  └── Release via standard process
```

Clarion, Stopwatch, and Lantern run continuously alongside Quill
throughout Phases 1–3. Wicket is always active.

## 10. Quality Gates — No Partial Implementation

The 0.5.1 release suffered from agents closing beads with work
deferred or half-done. This release enforces:

1. **Acceptance criteria on every bead** — extracted from specs, loaded
   via `bd update --acceptance`. This is the contract.

2. **Wicket runs `/review` on every PR** — checks against the bead's
   acceptance criteria and spec. Not just "compiles and passes tests"
   but "delivers what the spec promised."

3. **Mandatory harness tests** — every type system bead must include
   harness tests demonstrating the feature works. Machine-verifiable,
   not just Wicket's judgement.

4. **No bead closure without full verification** — coordinator closes
   beads after Wicket confirms all criteria met. No partial closure.
   No "defer remaining work to a follow-up." The bead is complete or
   it remains open.

5. **Wicket must not rubber-stamp** — Wicket's value is catching
   problems. A "looks good" review with no substantive comments is a
   failure of the role. Every review must demonstrate that the
   reviewer read the code, checked edge cases, and verified acceptance
   criteria.
