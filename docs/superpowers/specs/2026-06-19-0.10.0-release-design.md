# 0.10.0 Release Design — The Runtime: GC & Demand

**Date**: 2026-06-19
**Epic**: eu-9tah
**Status**: Draft

## Motivation

Eucalypt 0.9.x shipped with a working Immix mark-sweep GC with selective
evacuation, foundational demand annotations, and restored correctness after
several GC bug fixes. However, the runtime still has significant headroom:

- **No generational collection**: every GC is a full-heap traverse. Long-running
  programs (AoC solutions, IO pipelines) pay disproportionate pause costs.
- **Heuristic-only thunk management**: `suppress_update`, `single_use_args`, and
  hardcoded `strict` vectors are per-intrinsic hacks rather than
  analysis-driven. The IF intrinsic's then/else branches are hardcoded as
  single-use; tail-recursive conditionals need a runtime flag to avoid O(N)
  update frame accumulation.
- **Namespace functions are opaque to the inliner**: `str.split-on`, `cal.now`
  etc. are accessed via runtime `Lookup(namespace_var, "name")`, hiding the
  lambda from `distribute()` and preventing beta reduction.
- **Git imports lost in the Rust rewrite**: the documented `{ git: ..., commit:
  ..., import: ... }` syntax is silently ignored.

### Wave 1: Baseline + spec writing

- **Stopwatch** records baseline statistics for all AoC examples under
  `eu --statistics`: timings, allocation counts, GC counts. Committed as data
  files. No spec required.
- **Coordinator** writes individual specs for all remaining items (W10, W11,
  eu-398r, W12, eu-1gfj, eu-fwlt).

### Wave 2: Profiling + independent implementation

Runs in parallel:

- **Stopwatch** (eu-1gfj + eu-fwlt): allocation rate histograms (object sizes,
  env-frame vs other), GC frequency and pause duration, environment chain depth
  distribution. Results committed as data files.
- **Quill** (eu-398r): namespace lambda hoisting — new core pass extracting
  member lambdas from namespace blocks into top-level `Let` bindings, exposing

- **Lantern** (eu-9tah.3): git import restoration — `Locator::Git` variant,

- **Wicket** reviews and merges Quill and Lantern PRs as they land.
- **Coordinator** incorporates profiling results into W10 and W11 specs.

After Lantern's W12 PR merges, Lantern transitions to proactive tooling/docs
mode.

### Wave 3: GC + demand analysis

**Furnace** takes both sequentially. W10 first — W11 benefits from reduced GC
noise when measuring thunk elimination impact.

## Design: W10 — Sticky Immix Generational Nursery

### Strategy

Sticky Immix: young objects are unmarked objects in existing Immix blocks. No
separate nursery space, no copying for promotion.

### Nursery collection

A nursery collection marks reachable young (unmarked) objects, promoting them.
Unreachable young objects are reclaimed. Old (already marked) objects are not
traversed except as roots.

### Write barrier

Card table — one byte per 512-byte card. On any pointer store into an
already-marked (old) object, dirty the card. Nursery collection scans dirty
cards as additional roots.

**Mutation sites requiring the barrier:**
- `EnvFrame::update` (thunk update) — the primary site
- Block merge operations
- List cons operations
- Any other pointer writes into old objects

### Collection policy

- Nursery collect when the bump allocator exhausts the current block
- Full collect after N nursery collections or when fragmentation threshold is hit
- Tuning informed by Stopwatch's profiling data (wave 2)

### Acceptance criteria

- Nursery collections are faster than full collections on long-running AoC
  examples
- Total GC time reduced compared to eu-9tah.4 baseline
- No correctness regressions: full test suite passes
- `EU_GC_VERIFY=2` passes (structural verification)
- `EU_GC_POISON=1` passes (use-after-free detection)

## Design: W11 — Demand Analysis with Per-Function Signatures

### New core pass

Located in `src/core/`, runs after inline/fuse, before STG compile.

### Lattice

- **Strictness**: `{L, S}` (lazy, strict)
- **Cardinality**: `{Absent, AtMostOnce, Multi}` (Absent applies to unused
  function arguments — dead bindings are already removed by the prune pass)
- **Demand** = `(Strictness, Cardinality)` per binding/argument

### Analysis

Backward abstract interpretation over core expressions:

- `Case` scrutinee: strict
- Application: propagates callee's demand signature to arguments
- `Let`/`LetRec`: fixed-point iteration bounded by lattice height (finite, converges)
- Lambda bodies: collected into per-function demand signatures

### Per-function demand signatures

Each function gets a demand signature — a vector of demands, one per argument.
Stored on the core `Lam` node. Flows through call sites so that at each
application, the compiler knows the callee's per-argument demands without
looking inside.

### Removals

The following hacks are deleted, replaced by inferred demands:

- `suppress_update` flag in `StgSyn::Case` and the runtime check in
  `vm.rs:return_data()`
- `single_use_args()` trait method on `StgIntrinsic` and all implementations
  (IF's `[1, 2]`, etc.)
- Hardcoded `strict` vectors in `src/eval/intrinsics.rs`

### Acceptance criteria

- `suppress_update`, `single_use_args`, and `strict` vectors are deleted from
  the codebase
- IF-heavy programs (AoC conditionals, recursive patterns) show no performance
  regression
- Thunk allocation count reduced compared to eu-9tah.4 baseline
- `eu dump` shows demand annotations on bindings
- All tests pass, including error tests

## Design: eu-398r — Namespace Lambda Hoisting

### New core pass

Located in `src/core/`, runs between inline/fuse and STG compile (or early
enough for `distribute()` to see the results).

### Mechanism

1. Walk namespace block definitions (e.g. `str`, `cal`, `io`)
2. Extract member lambdas into top-level `Let` bindings with mangled names
   (e.g. `str.split-on` → `__str_split_on`)
3. Replace `Lookup(namespace_var, "name")` with direct variable references where
   the member is a known lambda
4. `distribute()` then sees bare `Lam` nodes and can inline them

### Acceptance criteria

- Namespace functions appear in `eu dump inlined` output as inlined expressions
- Measurable reduction in runtime lookup overhead on programs using namespace
  functions heavily
- No behavioural change: full test suite passes
- No change to user-visible syntax or semantics

## Design: eu-9tah.3 (W12) — Restore Git Imports

### Documented syntax (restoring prior behaviour)

```eu
{ import: { git: "https://github.com/user/repo"
            commit: "0140232cf882a922bdd67b520ed56f0cddbd0637"
            import: "path/to/file.eu" } }
```

- `git`: repository URL (any format git CLI accepts)
- `commit`: SHA — mandatory for reproducibility
- `import`: path within the repository

Multiple git imports mix freely with local imports in import lists.

### Implementation

1. Add `Locator::Git { url: String, commit: String, path: String }` variant
2. Pre-fetch step in source loader:
   - `git clone --bare <url>` to `~/.eu/cache/git/<url-hash>/`
   - `git archive <commit> -- <path>` to extract the target file
   - Cache keyed by `(url, commit)` — immutable, never re-fetched
3. Resolved filesystem path feeds into existing `load_tree()` / `read_to_core()`

### Error handling

- Missing `commit` field: error with clear message ("git imports require a
  commit SHA for reproducibility")
- Clone failure: error with URL and git stderr
- Missing path within repo: error naming the path and commit
- Network unavailable with cache hit: succeed from cache

### Acceptance criteria

- `{ import: { git: ..., commit: ..., import: ... } }` works end-to-end
- Cached imports do not re-fetch
- Clear errors on missing commit, bad URL, missing path
- Works with `.eu` files and data format imports (YAML, JSON, etc.)
- Full test suite passes

## Design: eu-1gfj + eu-fwlt — Profiling

### eu-1gfj: Profile env-frame allocation hot path

Instrument the bump allocator to measure:
- Allocation rate (allocations per second)
- Size distribution (histogram of object sizes)
- Env-frame allocations vs other allocations (proportion)
- Peak allocation pressure (max allocations between GC cycles)

### eu-fwlt: Benchmark O(depth) environment traversal

Measure environment chain traversal:
- Lookup chain depth distribution across representative programs
- Time spent in environment traversal vs total execution time
- Worst-case depth in AoC examples

### Acceptance criteria (both)

- Results committed as data files under a suitable location
- Results include AoC examples and at least one stress-test program
- Data is machine-readable (JSON or CSV) for comparison with post-0.10.0
  measurements

## Operating Model

### Agent roles

| Agent | Role | Merge authority |
|-------|------|-----------------|
| Coordinator | Write specs, validate completions, sequence waves | None |
| Furnace | Implement W10, W11 | PR only |
| Quill | Implement eu-398r | PR only |
| Lantern | Implement W12, then proactive tooling/docs | PR only |
| Stopwatch | Baseline stats, profiling, then proactive perf | PR only |
| Clarion | Proactive error diagnostics | PR only |
| Wicket | Gatekeeper | Merges implementation PRs; recommends proactive PRs |

### Gatekeeper protocol

- Wicket merges Furnace, Quill, Lantern implementation PRs directly to master
- Wicket closes beads only after verifying the spec is completely implemented
- CI must be green before any merge
- Proactive PRs (Clarion, Stopwatch, Lantern-proactive): Wicket reviews and
  recommends; owner merges
- Never merge when master CI is red (except targeted fixes)

### Spec-first rule

Every bead requires an approved spec before implementation begins. Exception:
eu-9tah.4 (baseline stats) which is preparatory data collection.

### Bead closure policy

Beads are closed only by Wicket after verifying that the implementation
satisfies all acceptance criteria in the spec. Close reason must reference
specific spec sections completed. Batch closure without verification is
forbidden.
