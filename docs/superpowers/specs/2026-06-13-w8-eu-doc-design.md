# W8: `eu doc` — Documentation & Schema Extraction

- **Date:** 2026-06-13
- **Status:** Draft
- **Bead:** eu-kgsi.3
- **Roadmap:** W8 — `eu doc` — documentation & schema extraction

---

## 1. Problem

The prelude reference is generated by a Python script (`scripts/extract-prelude-docs.py`)
that regex-parses `lib/prelude.eu`. It has no access to the parsed AST, the
type checker's alias resolution, or deprecation/visibility metadata. It is
fragile (regex-based), slow to evolve, and disconnected from the pipeline
that actually understands eucalypt source.

There is no way to extract documentation or a schema from an arbitrary `.eu`
unit. Users writing libraries have no tooling to generate API docs from their
`doc:`, `type:`, and `deprecated:` metadata.

## 2. Goal

A `eu doc` subcommand that extracts documentation and schemas from any `.eu`
unit using the real front-end pipeline — parse, desugar, type-check — and
outputs Markdown or JSON Schema. Replaces the Python script entirely for
prelude reference generation.

## 3. Design

### 3.1 Subcommand

```
eu doc <file.eu>                    # Markdown docs for a unit
eu doc <file.eu> --format json      # JSON Schema output
eu doc --prelude                    # Full prelude reference (replaces Python script)
eu doc --prelude --format json      # Prelude as JSON Schema
eu doc <file.eu> --check            # Coverage report (documented vs undocumented)
```

### 3.2 Extraction pipeline

`eu doc` runs the front-end pipeline up to type-checking but does **not**
evaluate:

```
parse → desugar → cook → eliminate → type-check → extract → render
```

Extraction reads from two sources:

1. **The desugared/cooked core expression** — binding names, structure,
   namespace groupings (`ns: { ... }` blocks).
2. **The type checker's state** — resolved type annotations, `type-def`/
   `result-def` aliases, `PreludeSummary` schemes.

This reuses the symbol-table machinery the LSP already has
(`src/driver/lsp/mod.rs` symbol table construction).

### 3.3 What is extracted per binding

```rust
struct DocEntry {
    /// Binding name (may be qualified: "str.trim").
    name: String,
    /// Namespace prefix if inside a namespace block.
    namespace: Option<String>,
    /// Doc string from ` "..." or ` { doc: "..." } metadata.
    doc: Option<String>,
    /// Type annotation from ` { type: "..." } metadata.
    type_annotation: Option<String>,
    /// Resolved type from the type checker (may be richer than annotation).
    synthesised_type: Option<Type>,
    /// Type aliases registered via type-def / result-def.
    type_aliases: Vec<(String, Type)>,
    /// Deprecation status.
    deprecated: Option<DeprecationInfo>,
    /// Visibility (public, internal, suppress).
    visibility: Visibility,
    /// Source location for linking.
    source_span: Option<Span>,
    /// Example code from ` { example: "..." } metadata.
    example: Option<String>,
    /// See-also references from ` { see-also: [...] } metadata.
    see_also: Vec<String>,
    /// Section heading (from ## comments above the binding).
    section: Option<String>,
}

struct DeprecationInfo {
    message: Option<String>,
    replaced_by: Option<String>,
}
```

### 3.4 Metadata sources

The desugarer preserves metadata as `Expr::Meta` nodes. Extraction walks
the desugared expression and reads:

| Metadata key | Source | Used for |
|---|---|---|
| `doc` | `` ` "text" `` or `` ` { doc: "text" } `` | Description |
| `type` | `` ` { type: "T" } `` | Type annotation |
| `type-def` | `` ` :type-def `` or `` ` { type-def: true } `` | Type alias |
| `result-def` | `` ` { result-def: true } `` | Return-type alias |
| `deprecated` | `` ` :deprecated `` or `` ` { deprecated: "msg" } `` | Deprecation |
| `replaced-by` | `` ` { replaced-by: "name" } `` | Migration path |
| `export` | `` ` :internal `` or `` ` { export: :suppress } `` | Visibility |
| `example` | `` ` { example: "code" } `` | Example code |
| `see-also` | `` ` { see-also: [:a, :b] } `` | Cross-references |

Section headings are extracted from `##` comments preceding declarations.
These are available in the Rowan parse tree as `COMMENT` tokens.

### 3.5 Markdown output

For a general unit, `eu doc file.eu` produces:

```markdown
# file

> Description from the unit's metadata string.

## Section Name

### `function-name`

```
type: number -> string
```

Description from doc metadata.

**Deprecated:** message. Use `replacement` instead.

**Example:**
```eu
function-name(42)
//=> "42"
```

**See also:** `other-fn`, `another-fn`
```

Bindings with `export: :internal` or `export: :suppress` are excluded.
Deprecated bindings are included with a deprecation notice.

### 3.6 JSON Schema output

`eu doc file.eu --format json` produces a JSON Schema document derived
from the unit's `type:` annotations and `type-def` aliases. The type DSL
maps to JSON Schema:

| Type DSL | JSON Schema |
|---|---|
| `string` | `{ "type": "string" }` |
| `number` | `{ "type": "number" }` |
| `symbol` | `{ "type": "string" }` |
| `null` | `{ "type": "null" }` |
| `[a]` | `{ "type": "array", "items": ... }` |
| `{{x: T, y: U}}` | `{ "type": "object", "properties": ..., "required": [...] }` |
| `T \| U` | `{ "oneOf": [...] }` |
| `"literal"` | `{ "const": "literal" }` |
| `a` (type variable) | `{}` (any) |

Functions (`T -> U`) are not representable in JSON Schema and are omitted
from the schema output (they appear only in the Markdown docs).

The schema covers the unit's **data declarations** — bindings whose types
describe data shapes rather than functions. This is the foundation W22
(host-language schema interop) builds on.

### 3.7 Prelude mode

`eu doc --prelude` generates the full prelude reference, replacing
`scripts/extract-prelude-docs.py`:

1. Loads the embedded prelude source.
2. Runs extraction as above.
3. Applies **category grouping** — maps sections and namespaces to
   categories (lists, blocks, strings, numbers, booleans, combinators,
   calendar, sets, random, metadata, io). The mapping currently lives in
   the Python script; it moves to a data file or is derived from the
   section/namespace structure.
4. Merges **supplement files** from `docs/reference/prelude/supplements/`.
   Each supplement adds extended prose, examples, or context to a function's
   entry. Supplements are keyed by `{category}/{slug}.md`.
5. Generates one Markdown file per category plus an index.
6. Reports **coverage** — documented vs undocumented public bindings.

Output goes to `docs/reference/prelude/` (the same location the Python
script writes to).

### 3.8 Doctest mode (future)

The roadmap mentions a doctest mode that runs examples in `doc:` metadata
under the test harness. This is deferred — the existing
`scripts/test-doc-examples.py` covers markdown-embedded examples. When
`example:` metadata becomes common, a `eu doc --test` flag can extract
and run them.

### 3.9 LSP integration

The extraction data feeds LSP features:

- **Hover**: show the `doc:` string and resolved type on hover (already
  partially implemented).
- **Completion**: include doc strings in completion items.
- **Go-to-definition**: source spans enable navigation.

This is not new work — the LSP already reads metadata. `eu doc` reuses
the same extraction logic so they stay in sync.

## 4. Implementation sketch

### Phase 1: Extraction core
- Define `DocEntry` and extraction logic.
- Walk the desugared expression, read metadata, collect entries.
- Extract section headings from Rowan `COMMENT` tokens.
- Handle namespaces (qualified names).
- Integrate type checker results (resolved types, aliases).

### Phase 2: Markdown renderer
- Render `Vec<DocEntry>` to Markdown.
- Handle sections, type annotations, deprecation notices, examples,
  see-also links.
- Filter by visibility.

### Phase 3: JSON Schema renderer
- Map type DSL types to JSON Schema.
- Generate schema from data-typed bindings.
- Handle type aliases (`type-def` names become `$defs`).

### Phase 4: `eu doc` subcommand
- Wire extraction + rendering into a new subcommand in `src/driver/`.
- `--format md|json` flag (default: md).
- `--check` flag for coverage reporting.
- File argument for general units.

### Phase 5: Prelude mode
- `--prelude` flag loads embedded prelude.
- Category mapping (from Python script's `SECTION_TO_CATEGORY` etc.).
- Supplement file merging.
- Multi-file output to `docs/reference/prelude/`.
- Coverage report.

### Phase 6: Retire Python script
- Verify output parity between `eu doc --prelude` and the Python script.
- Remove `scripts/extract-prelude-docs.py`.
- Update CI/docs build to use `eu doc --prelude`.

### Phase 7: Validation
- Output parity with the current generated reference.
- JSON Schema validates against the JSON Schema meta-schema.
- Coverage report matches Python script's numbers.
- `eu doc` on harness test files produces reasonable output.

## 5. Dependencies

- **s"..." (eu-kgsi.5)**: type-data literals allow schemas to reference
  types as values. `eu doc` can extract `s"..."` values from bindings
  for schema generation. Not strictly required for Markdown output but
  needed for the JSON Schema path.
- **W6 (pre-compiled prelude)**: `eu doc --prelude` can use the prelude
  blob's `PreludeSummary` for type information, avoiding a full
  type-check pass. Not required — `eu doc` works with source prelude too.

## 6. Test plan

- **Output parity**: `eu doc --prelude` generates reference pages matching
  the current Python-generated content (modulo formatting improvements).
- **Coverage**: `eu doc --prelude --check` reports the same documented/
  undocumented counts as the Python script's `--check` mode.
- **General unit**: `eu doc tests/harness/001_basic.eu` produces
  reasonable output.
- **JSON Schema**: `eu doc file.eu --format json` produces valid JSON
  Schema that validates against the meta-schema.
- **Type aliases**: `type-def` and `result-def` aliases appear in both
  Markdown and JSON Schema output.
- **Deprecation**: deprecated bindings show deprecation notices and
  `replaced-by` suggestions.
- **Visibility**: `export: :internal` and `export: :suppress` bindings
  are excluded.
- **Supplements**: supplement files are merged correctly into the
  prelude reference.

## 7. Risks

- **Category mapping fragility**: the Python script's category mapping is
  hard-coded. Moving it to `eu doc` means maintaining it in Rust. Mitigate:
  derive categories from the prelude's own section/namespace structure
  rather than maintaining a parallel mapping.
- **Type-to-JSON-Schema mapping**: the eucalypt type DSL is richer than
  JSON Schema (recursive types, type variables, higher-kinded types).
  Mitigate: map what maps cleanly, skip what doesn't (functions, HKT),
  document the subset.
- **Output format stability**: generated docs become part of the published
  reference. Mitigate: use the conformance/golden approach — generate,
  bless, diff on changes.

## 8. Success criteria

- **`scripts/extract-prelude-docs.py` is deleted** and `eu doc --prelude`
  generates equivalent or better reference pages.
- **Any `.eu` file** can have its documentation extracted with `eu doc`.
- **JSON Schema output** is valid and covers data-typed bindings.
- **The prelude reference is generated from the prelude itself** — no
  hand-written duplication.
