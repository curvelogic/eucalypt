# Vec Type for O(1) Indexed Access

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **MANDATORY**: Before writing ANY eucalypt (.eu) code, read: `docs/reference/agent-reference.md`, `docs/appendices/syntax-gotchas.md`, `docs/appendices/cheat-sheet.md`

**Goal:** Add a `vec` type backed by `Vec<Primitive>` for O(1) indexed access, random sampling, and slicing on large primitive collections. Vec is an opt-in type with its own `vec.` namespace, following the pattern of `set.` and `arr.`.

**Bead:** eu-v5n1

**Motivation:** Cons-lists are O(n) for indexed access. Random sampling 10 elements from 10,000 requires 10 × O(n) walks = O(100,000) closure forcings. With a vec, it's 10 × O(1) = O(10). The user opts in with `vec.of(list)` when they know they need indexed access.

**Architecture:** Follows the existing `HeapSet` / `HeapNdArray` pattern:
- `HeapVec` in `src/eval/memory/` — heap-allocated `Vec<Primitive>`
- `Native::Vec(RefPtr<HeapVec>)` variant
- Intrinsics in `src/eval/stg/vec.rs` — `vec.` namespace functions
- Prelude bindings in `lib/prelude.eu`

**Tech Stack:** Rust

- [ ] **Step 1: Create `HeapVec`**

```rust
use super::alloc::StgObject;
use super::set::Primitive;  // reuse the same Primitive type as HeapSet

/// A vector of primitive values, stored on the heap.
///
/// GC does not trace into vec contents because primitives contain no
/// heap references (strings are owned, symbols are interned IDs).
#[derive(Debug, Clone)]
pub struct HeapVec {
    elements: Vec<Primitive>,
}

impl StgObject for HeapVec {}

impl HeapVec {
    pub fn empty() -> Self {
        HeapVec { elements: Vec::new() }
    }

    pub fn from_primitives(iter: impl Iterator<Item = Primitive>) -> Self {
        HeapVec { elements: iter.collect() }
    }

    pub fn len(&self) -> usize {
        self.elements.len()
    }

    pub fn is_empty(&self) -> bool {
        self.elements.is_empty()
    }

    pub fn get(&self, index: usize) -> Option<&Primitive> {
        self.elements.get(index)
    }

    pub fn slice(&self, from: usize, to: usize) -> Self {
        let from = from.min(self.elements.len());
        let to = to.min(self.elements.len());
        HeapVec {
            elements: self.elements[from..to].to_vec(),
        }
    }

    pub fn elements(&self) -> &[Primitive] {
        &self.elements
    }
}
```

```rust
pub fn machine_return_vec(
    machine: &mut dyn IntrinsicMachine,
    view: MutatorHeapView<'_>,
    vec: HeapVec,
) -> Result<(), ExecutionError> {
    let ptr = view.alloc(vec)?;
    machine.set_native(Native::Vec(ptr));
    Ok(())
}
```

## Chunk 2: Vec Intrinsics

### Task 2: Implement vec namespace intrinsics

**Files:**
- Create: `src/eval/stg/vec.rs`
- Modify: `src/eval/stg/mod.rs` (register module and intrinsics)
- Modify: `src/eval/intrinsics.rs` (add to catalogue)

- [ ] **Step 1: `VecOf` — convert list to vec**

`VEC_OF(list)` — forces the entire list, collects primitives into a `HeapVec`.

Uses the existing `data_list_arg` iterator to walk the cons-list, extracting each element as a `Primitive`. Returns `Native::Vec`.

This is O(n) — the user is opting in by calling `vec.of`.

- [ ] **Step 2: `VecLen` — O(1) length**

`VEC_LEN(vec)` — returns `HeapVec::len()` as a number.

- [ ] **Step 3: `VecNth` — O(1) indexed access**

`VEC_NTH(n, vec)` — returns `HeapVec::get(n)` as a native. Panics (or returns error) if index out of bounds.

- [ ] **Step 4: `VecSlice` — O(1) sub-range**

`VEC_SLICE(from, to, vec)` — returns a new vec containing elements `[from, to)`. Clamps to bounds.

- [ ] **Step 5: `VecSample` — random sampling**

`VEC_SAMPLE(n, seed, vec)` — picks `n` random elements from the vec without replacement. Uses the existing SplitMix64 PRNG (from `src/eval/stg/prng.rs`).

Algorithm:
1. Generate `n` random indices using SplitMix64 (Fisher-Yates partial shuffle on index array)
2. Collect the elements at those indices
3. Return as a new vec (or list — design choice)

Returns a vec (caller can `vec.to-list` if needed).

- [ ] **Step 6: `VecShuffle` — Fisher-Yates shuffle**

`VEC_SHUFFLE(seed, vec)` — returns a new vec with elements in random order. Uses SplitMix64.

- [ ] **Step 7: `VecToList` — convert vec back to cons-list**

`VEC_TO_LIST(vec)` — builds a cons-list from the vec elements. O(n).

Uses `machine_return_str_list` or a similar helper adapted for mixed primitives.

- [ ] **Step 8: `VecMap` — apply function to each element**

`VEC_MAP(f, vec)` — applies `f` to each element, collects results into a new vec.

This is trickier than the pure-primitive intrinsics because `f` is a closure. The intrinsic would need to evaluate `f(element)` for each element and collect the results. This may need to be implemented as an STG wrapper rather than a pure execute() method, or deferred in favour of `vec.to-list map(f) vec.of`.

**Decision point:** If vec.map is too complex for the first version, defer it. Users can `vec.to-list` → pipeline → `vec.of` for transformation.

- [ ] **Step 9: Register all intrinsics**

Add to the intrinsic catalogue in `src/eval/stg/mod.rs` and `src/eval/intrinsics.rs`.

## Design Decisions

- **Vec holds `Primitive` only** — same as Set. No closures, no blocks, no nested lists. This keeps GC simple (no tracing) and matches the use case (large flat collections of strings/numbers).
- **`vec.of` is the only entry point** — string/import intrinsics continue to return cons-lists. Vec is opt-in.
- **`vec.map`/`vec.filter` deferred** — for the first version, users convert to list for pipeline operations. This avoids the complexity of evaluating closures across a vec in the intrinsic layer.
- **`vec.sample` returns a vec** — consistent type, caller can `vec.to-list` if needed.
- **Rendering as a list** — vecs should render identically to lists in output formats. They're a performance optimisation, not a semantic distinction.

## Future Considerations

- **`vec.map`/`vec.filter`** — worth adding later if vec-heavy workflows emerge
- **Direct vec-producing intrinsics** — `str.split-on` could offer a `str.split-on-to-vec` variant or a flag to skip cons-list construction
- **Lazy intrinsic returns** — separate issue for a later version; would benefit cons-lists too
- **Collection protocol (option 6)** — if the `vec.` namespace proliferates, consider a polymorphic abstraction
