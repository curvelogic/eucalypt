# HeapSyn 0.11.1 → 0.12 regression analysis (companion to eu-mhjz)

**Date:** 2026-07-03
**Author:** Stopwatch. **Status:** measurement + analysis only — no code changes.
**Baselines built clean (stable-aarch64, rustc 1.96.0):**
`0.11.1` tag @ 765b3906 (`/tmp/eu-011`, HeapSyn-only) vs `integration/0.12.0`
@ 185dda4a (`/tmp/eu-perf`, embedded blob regenerated).
All AoC/bench **source files and inputs are byte-identical between the two
trees** — the only variable is the binary (runtime + prelude).
Method: external wall-clock, 5-run medians, `--heap-limit-mib 12288`, `timeout`.

---

## Verdict (answers to the three questions)

1. **Is the day08 GC death-spiral pre-existing or new?**
   **New in 0.12 — but it is NOT a GC/engine change.** It is caused by the
   **BV5 embedded prelude blob (eu-amp9)**. day08-p1 runs in **5.15 s on 0.11.1**
   and **5.09 s on 0.12 with the source prelude**; it only spirals to **>500 s**
   on 0.12 with the **blob** prelude.

2. **Did we break the HeapSyn engine?**
   **No — the HeapSyn engine is unchanged.** On 0.12 with the source prelude,
   HeapSyn matches 0.11.1 on every benchmark (ratio 0.90–1.02). The regression
   is entirely the **blob-compiled prelude**, which is 1.4–2.5× slower (∞ on
   day08) than the source-compiled prelude — and it slows **bytecode too**
   (day08 bytecode 7.6 s source → 9.9 s blob).

3. **Compare with previous AoC benchmarking.**
   The recorded 0.11.0 figures (`examples/aoc25/PERFORMANCE.md`) match 0.12
   **source-prelude** numbers (e.g. day03-p2 0.11.0 0.517 s ≈ 0.12-src 0.512 s;
   day09-p1 ≈ 4.1 s both). Those recorded numbers are the *source-prelude*
   reality; the blob path is a new, previously-unmeasured regression. Note the
   recorded day08 **alloc** figure (118 M) is stale — 0.11.1 measures 18.7 M.

## Mechanism: the blob prelude allocates more for the same work

`-S` allocation counts, HeapSyn, source vs blob:

| workload | ticks src → blob | allocs src → blob | alloc ×  |
|---|---|---|--:|
| day01-p1 | 12.96 M → 12.92 M (≈=) | 1.87 M → 2.24 M | **1.20×** |
| day08 `test` path | 800 k → 863 k (+8 %) | **82 k → 466 k** | **5.65×** |

**Near-identical tick counts but up to 5.65× more heap allocations.** Same work,
far more allocation ⇒ the blob path is **re-allocating structure the source path
shares** — i.e. lost prelude-global / CAF-memoisation (or lost prelude↔user
co-compilation inlining) in the pre-encoded blob. day08 leans hardest on the
affected construct, so it blows up most; at full scale (`solve-n(1000,150)`) the
5.6× allocation tips the HeapSyn managed heap into a GC death-spiral (mark cost
escalating — `mark_region_in_block` was the hot GC frame in the eu-mhjz profile).
Bytecode routes transients through the C heap so it avoids the spiral, but still
pays ~1.3× for the extra allocation.

### Bisect note
`git bisect` on day08-p1 over 765b3906..185dda4a reported the final (docs-only)
merge as "first bad" — because the bisect builds do a plain `cargo build`
(stale blob ⇒ **source-prelude fallback**), so every code commit ran day08 in
~5.5 s. That itself is the proof: **no source commit regressed day08; only the
regenerated blob does.**

## Impact on the eu-mhjz A/B conclusions (important correction)

eu-mhjz ran **blob-vs-blob**, so its "HeapSyn" column was **handicapped by this
blob bug**, most severely on day08:

- The **"bytecode >52× win on day08" was not a bytecode dispatch win** — it was
  HeapSyn crippled by the blob regression. With the **source prelude**, day08
  HeapSyn (5.1 s) is actually **faster than bytecode** (source 7.6 s, blob 9.9 s).
- The other allocation-heavy A/B ratios are partly blob-contaminated on both
  engines. **A fair engine A/B should be re-run source-vs-source (or after the
  blob fix).** The pure-dispatch parity finding (day11-p1) is unaffected (blob
  ≈ source there).

## Recommendations (for owner — not implemented)

1. **Fix the BV5 blob prelude to preserve the sharing/inlining the source
   pipeline achieves** (eu-amp9). Diff the blob-compiled vs source-compiled
   prelude STG for the constructs day08 hammers (indexing `!!`, `map`, `first`,
   `apply`, the x-sort/candidate path); the tell is 5.6× allocation at equal
   ticks — look for a prelude global/CAF that is shared under source but rebuilt
   per-use under the blob. This is the single highest-value fix: it restores
   1.4–2.5× across allocation-heavy programs and removes the day08 spiral for
   **both** engines.
2. **Add a blob-vs-source allocation-parity check** to CI (assert `allocs` within
   a few % on a day08-class program) so a blob that loses sharing is caught
   automatically.
3. **Re-run eu-mhjz source-vs-source** for the clean engine verdict once (1)
   lands; until then treat the eu-mhjz day08 result as a blob artefact, not an
   engine result.
4. The HeapSyn engine needs no 0.11→0.12 remediation — it is unchanged.
