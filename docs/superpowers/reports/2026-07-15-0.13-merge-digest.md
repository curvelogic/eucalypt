# 0.13 pre-release merge digest and risk audit

**Scope:** every merge to `master` since the `0.12.1` tag (2026-07-13 00:23 +0100)
up to the current tip, `b553539b` (PR #1012, 2026-07-15 06:27 UTC).
**Author:** Wicket, at the owner's request, as a read-only audit (no production
code changed by this report).
**Method:** `git log --merges 0.12.1..origin/master` cross-referenced against

**Key findings**

1. **#1002 (predecode IR Phase 1) has zero recorded review** — the flagship
   0.13 feature's actual implementation PR (new `Instr` format, `unsafe`
   array-fill code, GC-adjacent dispatch rewrite) carries no Wicket review
   comment on GitHub and no review note in its bead (`eu-2sa6.13`), unlike
   every other PR in its own chain (#1004, #1009, #1010 all have thorough,
   independently-reproduced reviews). CI was green. I did an adversarial
   re-read and one fresh build-and-test spot check myself (below) to close
   this gap retroactively; the code holds up, but the process gap is real and
   should not recur. **Action: flag for the coordinator/owner.**
2. **PR #1010 (copy-specialisation of recursive higher-order combinators,
   `eu-dp0k`) is unconditional, default-on codegen** — `tag::tag_combinators`
   is called unconditionally from `src/driver/source.rs:774` (and the WASM
   pipeline), not behind a flag. It changes the compiled output of every
   program that recurses through a qualifying combinator shape (delivered
   −96% ticks on `022_hof_fold`). **This has no CHANGELOG entry.** Contrast
   with #1002/#1004/#1009 (predecode IR), which are correctly *absent* from
   CHANGELOG because they're flag-gated behind `EU_PREDECODE=1` and not yet
   default. #1010 is default and user-observable (compiled bytecode size,
   `xtask prelude-compile` inline-core counts change) and belongs in
   "Unreleased". **Action: needs a CHANGELOG entry before the 0.13 cut.**
3. **#1012 caught a real, pre-existing, user-facing bug via its own new
   test** — a genuine Windows bug (`Locator::from(&str)` misparsing any
   absolute Windows path, e.g. `eu C:\foo.eu`, as a URL instead of a
   filesystem path, RFC-3986 single-letter-scheme collision) was exposed by
   Windows CI failing on #1012's new Smid regression test, root-caused, fixed
   in production code (not just the test), and re-verified before merge. This
   is the review process working as intended — flagging it here only because
   it's exactly the "Windows locator" genre of hazard the audit brief asked
   to watch for, and it's now closed, not open.
4. **#1012 also caught and fixed a genuine correctness regression before
   merge**: an eval-path type-warning false negative (`065_lookup_key_typo_warns.eu`)
   that green CI didn't catch because the `typecheck/` harness only ever
   exercises `eu check --strict`, never plain `eu` eval — a real
   eval/check-path test-coverage gap. Wicket sent the PR back, Furnace fixed
   the regression, and added the coverage. The **process worked**, but the
   underlying coverage gap (`typecheck/` tests still don't assert the eval
   path by default) is only closed for the one file (065), not systemically.
   Worth a follow-up bead if not already filed.
5. **Three open pre-release blockers, already known and tracked, still open
   as of this writing**: `eu-rb5n` (source-path fusion closure — explicitly
   marked "0.13 RELEASE BLOCKER", `IN_PROGRESS`), `eu-2sa6.20` (dead
   `PreludeBlob::type_summary` removal, `OPEN`), `eu-2sa6.21` (docs phase 2,
   `OPEN`). None of these are digest findings — they were already flagged in
   PR review comments and filed as beads — but they are the concrete
   punch-list for what's left. See "Release readiness" below.
6. **No correctness or safety issues found in the deep-dive that aren't
   already tracked.** The `unsafe` block introduced in #1002's predecode
   `Op::LetRec` handler is a byte-for-byte structural port of the existing,
   already-shipped byte-path pattern (same pre-size/alloc/second-pass-fill
   shape, same safety invariant) — not a novel risk.

**Readiness verdict: not yet release-ready, but no undisclosed blockers.**
Everything currently known to be missing is already tracked in beads
(`eu-rb5n`, `eu-2sa6.20`, `eu-2sa6.21`) and owned. The one *new* item this
audit surfaces is the missing CHANGELOG entry for #1010's default codegen
change (finding 2), plus the process observation about #1002's unreviewed
merge (finding 1) for the coordinator to act on. Recommend closing those two
before cutting 0.13, on top of the three already-tracked blockers.

## Deep-dive: adversarial re-check of the HIGH-risk set

### #1002 — predecode IR Phase 1 (unreviewed at merge time)

This is the one PR in the batch with no documented review, so I re-read it
from scratch and ran an independent verification pass rather than just
reading the PR body.

- **`unsafe` code**: one `unsafe { array.set_unchecked(...) }` block is
  introduced in the new predecode-path `Op::LetRec` handler
  (`src/eval/bytecode/machine.rs`). I compared it line-for-line against the
  pre-existing byte-path `Op::LetRec` handler (same file, already shipped in
  0.12.x): identical shape — pre-size the array with dangling closures,
  allocate the env frame, then a second pass fills each slot via
  `set_unchecked` with the same `// SAFETY: array was pre-sized to N and i <
  N` invariant. This is a faithful structural port of an already-reviewed
  pattern, not a new unsafe surface.
- **GC invariant**: `predecode.rs`'s own module doc states the invariant
  plainly ("nothing in these structures is a GC pointer... the collector
  never scans any of it") and the `Instr` struct is a plain `Copy` record of
  scalars/ordinals (compile-time-asserted to 16 bytes) — no raw pointers, no
  transmutes, no other `unsafe` in the 641-line `predecode.rs` module.
- **Independent spot check (this audit)**: built `master` at the current tip
  from a clean `cargo build --release`, then ran the full harness suite under
  `EU_PREDECODE=1 EU_GC_VERIFY=2 EU_GC_POISON=1 EU_GC_STRESS=1 cargo test
  --release --test harness_test` — **498/498 passed**, exercising full
  structural GC verification (mark/line/evacuate/sweep/forwarding checks),
  poison-based use-after-free detection, and stress-forced collection on
  every predecode-path test. This corroborates the correctness claims that
  should have been independently checked at merge time.
- **Design-conformance deltas**: the PR body itself discloses three
  deviations from the owner-signed design (16-byte vs the design doc's
  internally-inconsistent 20-byte field list — resolved in the design's own
  favour per its §3 decision; eager-only decode vs the design's eager+lazy
  split — a correctness-neutral simplification, explicitly flagged as "delta
  to confirm"; and a wider static-peel helper surface than the design
  named). None of these look concerning on inspection, but they were
  **self-reported by the author with no independent sign-off**, which is
  exactly the kind of thing the review process exists to catch or ratify.
- **Bottom line**: I found nothing that changes the merge decision — the code
  is sound on this re-read and the independent GC-stress run passes clean —
  but the process gap (a GC-adjacent, `unsafe`-containing PR merged with zero
  recorded review, in a project whose gate explicitly names "GC or memory
  layout change" as owner/Wicket review territory) should be flagged to the
  coordinator regardless of the outcome.

### Cross-merge scan for silent behaviour changes

I looked for any wire-format, error-path, or evaluation-order change that
might have shipped without an explicit callout:

- **`BYTECODE_WIRE_FORMAT_VERSION` 2→3** (#1012): correctly bumped, blob
  regenerates automatically on version mismatch, confirmed consistent between
  `build.rs` and `xtask/src/main.rs`. No compat trap — v3 has not shipped in
  any release yet, so there's no format the field-order/type_summary changes
  could break in the field.
- **Error paths**: #1012's regression (finding 4) was exactly an error/warning-path
  behaviour change slipping past CI; it's now fixed and covered. No other PR
  in the batch touches error construction or message text in a way flagged
  by any reviewer or visible in the diffs I read.
- **Evaluation order**: #1002/#1004's predecode dispatch is a decode-strategy
  change only (ordinal-space vs byte-offset-space), not a reordering of
  evaluation; #1010's copy-specialisation only offers additional inline
  candidates to the existing distribute/beta-reduce passes, it doesn't change
  their fixed-point behaviour (confirmed in #1010's own review: "even a
  mistaken tag can't cause non-termination", "substitution still resolves via
  its already-assigned, correct de Bruijn index").
- **Platform-specific hazards**: the Windows locator bug (#1012, finding 3)
  is the one genuine instance in this batch, already caught and fixed before
  the merge that's in `master` today.

## CHANGELOG completeness check

`CHANGELOG.md`'s `[Unreleased]` section currently documents: the prefix-list
type (#1001/#993), the pre-inline type-diagnostics fix, the blob-baked
prelude-core eval/check parity fix and the Windows locator fix (all #1012),
the source-prelude tick-handicap tripwire (#998), and the `markup.eu`
accessor + `deconstruct_head` fixes (#999). All of these are accurate against
the merged code (spot-checked the Windows fix and the file-count/latency
numbers, which match the final, corrected #1012 CHANGELOG text exactly, not
the earlier overstated draft).

**Gap found**: **#1010 (`eu-dp0k`, copy-specialisation of recursive
higher-order combinators) has no CHANGELOG entry**, despite being unconditional
default-on codegen (see finding 2 above) with a measured −96% tick
improvement on higher-order-fold-heavy code. This is the one user-visible
default-behaviour change in the batch that CHANGELOG is silent on. Everything
else that's absent from CHANGELOG (predecode IR, the diagnostic env-var
instrumentation, the docs/design/report PRs) is correctly absent because it's
either flag-gated-and-not-yet-default, gated-and-off-by-default, or genuinely
not user-visible.

**Engine perf numbers rule**: I did not find any bare, unlabelled tick/wall
figure entering CHANGELOG from this batch. The predecode-IR and
copy-specialisation performance figures live only in `docs/superpowers/reports/`
and PR bodies (correctly, since neither feature is default/shipped yet in the
CHANGELOG sense for predecode, and #1010's CHANGELOG entry doesn't exist yet
to check). When #1010's entry is written, it should carry a confidence label
and cite `docs/superpowers/specs/2026-07-13-recursive-combinator-copy-specialisation-design.md`
and/or the `eu-98zg`/`eu-yuto` reports per the PROTOCOL rule, since its
headline numbers (022_hof_fold −96%, etc.) are exactly the kind of figure the
protocol is meant to gate.

## Release readiness

**Not release-ready yet — three already-tracked blockers remain open**,
confirmed via `bd show` in this audit:

1. **`eu-rb5n`** (source-path fusion closure) — explicitly titled "0.13
   RELEASE BLOCKER" in the bead, status `IN_PROGRESS`. Owner decision on
   record (2026-07-14): copy-specialisation lands blob-path-first (`eu-dp0k`,
   done via #1010); this bead must close the source-path
   (`EU_SOURCE_PRELUDE=1`) divergence before 0.13 ships. Two sub-problems
   identified and verified but not yet fixed: `distribute`'s Meta-blindness
   on the source path, and demand-specialisation's dependency on blob-only
   inline-core pre-expansion parity.
2. **`eu-2sa6.20`** (remove dead `PreludeBlob::type_summary`) — `OPEN`,
   P2. Wicket-verified as genuinely dead (write-only) during #1012's review.
   Correctly deferred out of #1012 itself (scope discipline), but flagged as
   "cheapest possible moment" to remove since wire-format v3 hasn't shipped
   yet — after 0.13 cuts it needs a v4 bump instead.
3. **`eu-2sa6.21`** (docs phase 2: architecture.md bytecode-engine rewrite,
   error-messages.md, reflect.eu guide coverage, 3 syntax-gotcha entries) —
   `OPEN`, P2, owner-approved scope for items 1-4 of the docs-review plan
   (items 5-9 deferred to 0.14).

**Two additional items this audit surfaces, beyond the known set**:

4. **#1010 needs a CHANGELOG entry** before the cut (finding 2) — currently
   the only default, user-observable codegen change in the batch with no
   CHANGELOG coverage.
5. **Coordinator/process note on #1002** (finding 1) — no action needed on
   the code itself (independently verified clean in this audit), but the
   review-and-merge process should confirm why a GC-adjacent, `unsafe`-containing
   PR skipped documented review, so it doesn't recur on the next

Everything else audited in this digest — CI status, design-conformance,
GC safety, wire-format handling, and the four HIGH-risk diffs — checks out
clean.
